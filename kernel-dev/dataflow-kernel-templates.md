# Dataflow Kernel Templates for Compiler-Generated Ops

Hardcoded reader/writer templates that, combined with generated SFPI compute kernels, cover the full op space. Derived from tinygrad's `sops.gz` linearizer dump (4,457 kernels) and tt-metal CB conventions.

## Why hardcode dataflow kernels?

Dataflow kernels (reader on NCRISC, writer on BRISC) are boilerplate: read tiles from DRAM into CBs, write tiles from CBs to DRAM. The actual compute lives in the TRISC compute kernel (SFPI). A compiler should **generate** compute kernels and **select** from a fixed set of dataflow templates.

## What tinygrad's linearizer actually outputs

Analysis of `tinygrad/extra/datasets/sops.gz` (4,457 kernels from beam search dataset):

### Arity distribution (DEFINE_GLOBAL = number of buffer args)

| Globals | Kernels | % | Pattern |
|---------|---------|---|---------|
| 1 | 661 | 14.8% | Const fill / PRNG (0 in, 1 out) |
| 2 | 1,860 | 41.7% | **Unary** (1 in, 1 out) |
| 3 | 828 | 18.6% | **Binary** (2 in, 1 out) |
| 4 | 441 | 9.9% | **Ternary** (3 in, 1 out) |
| 5 | 234 | 5.3% | Fused 4-input |
| 6+ | 433 | 9.7% | Heavily fused (up to 24 globals) |

**75% of kernels are unary, binary, or const-fill.** 95% have ≤5 inputs.

### Outputs: almost always 1

- 4,455 kernels have exactly 1 STORE
- 2 kernels have 2 STOREs

Single-output writer covers 99.95% of cases.

### Reduction prevalence

45.7% of kernels contain REDUCE_AXIS. Reductions need a different reader pattern (the reader must loop K times per output tile, not 1:1).

### Compute ops used

Dominant: MUL (8,776), ADD (7,242), WHERE (2,163), CMPNE (1,860), CAST (1,642), REDUCE_AXIS (2,051).
Transcendentals: RECIP (573), MAX (540), SQRT (422), EXP2 (382), LOG2 (306), SIN (248).
Rare: THREEFRY (136), BITCAST (77), XOR (74), POW (39), IDIV (153), MOD (8).

All of these map to SFPI instructions or short SFPI sequences. The dataflow is orthogonal.

## The template set

### Template 1: Const-fill writer (no reader)

No reader kernel needed. Compute generates tiles from constants/PRNG, writer drains them.

Covers: 661 kernels (14.8%) — zero-fill, constant tensors, arange, random.

```
Pipeline:  [no reader] → compute(SFPI) → writer
CBs used:  c_16 (output only)
```

No reader kernel at all. Compute kernel produces tiles directly:

```cpp
// compute kernel (TRISC) — const fill
void kernel_main() {
    constexpr uint32_t cb_out = tt::CBIndex::c_16;
    uint32_t n_tiles = get_arg_val<uint32_t>(0);

    for (uint32_t i = 0; i < n_tiles; ++i) {
        tile_regs_acquire();
        // SFPI: fill dst_reg with constant value
        // (generated by compiler)
        tile_regs_commit();
        tile_regs_wait();
        cb_reserve_back(cb_out, 1);
        pack_tile(0, cb_out);
        cb_push_back(cb_out, 1);
        tile_regs_release();
    }
}
```

Writer: use **Template W1** below.

### Template 2: N-ary elementwise reader

A single parameterized reader that handles 1..N input buffers. Every elementwise op (unary, binary, ternary, fused N-input) uses the same reader — just with a different `num_inputs` runtime arg.

Covers: 3,570 kernels (80%) — all non-reduction, non-const ops.

```
Pipeline:  reader(N inputs) → compute(SFPI) → writer
CBs used:  c_0..c_{N-1} (inputs), c_16 (output)
```

```cpp
// reader kernel (NCRISC) — N-ary elementwise
// Runtime args: num_inputs, n_tiles, then N×(addr, dtype) pairs
void kernel_main() {
    uint32_t arg_idx = 0;
    uint32_t num_inputs = get_arg_val<uint32_t>(arg_idx++);
    uint32_t n_tiles   = get_arg_val<uint32_t>(arg_idx++);

    // Build address generators for each input
    // Max 8 inputs (c_0..c_7), practically ≤6
    InterleavedAddrGenFast<true> addrgen[8];
    uint32_t cb_ids[8];
    for (uint32_t k = 0; k < num_inputs; ++k) {
        uint32_t addr = get_arg_val<uint32_t>(arg_idx++);
        cb_ids[k] = tt::CBIndex::c_0 + k;
        uint32_t tile_bytes = get_tile_size(cb_ids[k]);
        addrgen[k] = {
            .bank_base_address = addr,
            .page_size = tile_bytes,
            .data_format = get_dataformat(cb_ids[k]),  // from CB config
        };
    }

    // Stream tiles: all inputs are read in lockstep (same tile index)
    for (uint32_t i = 0; i < n_tiles; ++i) {
        for (uint32_t k = 0; k < num_inputs; ++k) {
            cb_reserve_back(cb_ids[k], 1);
            noc_async_read_tile(i, addrgen[k], get_write_ptr(cb_ids[k]));
        }
        noc_async_read_barrier();
        for (uint32_t k = 0; k < num_inputs; ++k) {
            cb_push_back(cb_ids[k], 1);
        }
    }
}
```

When `num_inputs == 1`, this is the unary reader. When `num_inputs == 2`, binary. No need for separate templates — the inner loop is trivial and the branch on N is a runtime arg check, not a code generation concern.

**Note on heavily-fused kernels (6+ inputs):** The tinygrad data shows kernels with up to 24 global buffers. These are legitimate — a single reduction kernel might fuse 12 small weight tensors. The N-ary reader handles this naturally. The CB limit is 8 inputs (`c_0..c_7`), so kernels with >8 inputs need CB reuse (read input K into a CB, compute consumes it, then reuse that CB for input K+8). This is a compiler scheduling concern, not a dataflow template concern.

### Template 3: Reduction reader

Reductions need a different access pattern: for each output tile, the reader must stream K input tiles (the reduction dimension). The reader loops `n_tiles_out × K` total reads, but the compute kernel consumes K tiles per output.

Covers: ~2,038 kernels (45.7%) — sum, max, mean, variance, matmul-like patterns.

```
Pipeline:  reader(1+ inputs, K-loop) → compute(SFPI reduce) → writer
CBs used:  c_0 (reduced input), optionally c_1..c_{N-1} (other inputs), c_16 (output)
```

```cpp
// reader kernel (NCRISC) — reduction along one axis
// Runtime args: n_tiles_out, reduce_dim_tiles (K), input addr(s)
void kernel_main() {
    uint32_t arg_idx = 0;
    uint32_t n_tiles_out = get_arg_val<uint32_t>(arg_idx++);
    uint32_t K           = get_arg_val<uint32_t>(arg_idx++);
    uint32_t in_addr     = get_arg_val<uint32_t>(arg_idx++);

    constexpr uint32_t cb_in = tt::CBIndex::c_0;
    const uint32_t tile_bytes = get_tile_size(cb_in);

    const InterleavedAddrGenFast<true> in = {
        .bank_base_address = in_addr,
        .page_size = tile_bytes,
        .data_format = get_dataformat(cb_in),
    };

    uint32_t tile_idx = 0;
    for (uint32_t out = 0; out < n_tiles_out; ++out) {
        for (uint32_t k = 0; k < K; ++k) {
            cb_reserve_back(cb_in, 1);
            noc_async_read_tile(tile_idx++, in, get_write_ptr(cb_in));
            noc_async_read_barrier();
            cb_push_back(cb_in, 1);
        }
    }
}
```

**Reduction + extra inputs** (e.g., `reduce(a * b, axis=0)`): The reduction reader becomes an N-ary reduction reader. The reduced input streams K tiles per output; the other inputs stream 1 tile per output (broadcast along the reduce dim). This is a straightforward extension — add the N-ary loop from Template 2 inside the K loop for the broadcast inputs.

### Template W1: Single-output writer

One writer template covers 99.95% of kernels.

```cpp
// writer kernel (BRISC) — single output
void kernel_main() {
    uint32_t out_addr = get_arg_val<uint32_t>(0);
    uint32_t n_tiles  = get_arg_val<uint32_t>(1);

    constexpr uint32_t cb_out = tt::CBIndex::c_16;
    const uint32_t tile_bytes = get_tile_size(cb_out);

    const InterleavedAddrGenFast<true> out = {
        .bank_base_address = out_addr,
        .page_size = tile_bytes,
        .data_format = get_dataformat(cb_out),
    };

    for (uint32_t i = 0; i < n_tiles; ++i) {
        cb_wait_front(cb_out, 1);
        noc_async_write_tile(i, out, get_read_ptr(cb_out));
        noc_async_write_barrier();
        cb_pop_front(cb_out, 1);
    }
}
```

## Summary: 3 readers + 1 writer = full coverage

| Template | Reader | Inputs | Covers | % of sops.gz |
|----------|--------|--------|--------|-------------|
| Const-fill | (none) | 0 | const, arange, PRNG | 14.8% |
| N-ary eltwise | N-ary reader | 1–8 | all elementwise (unary/binary/ternary/fused) | 39.5% |
| Reduction | Reduce reader | 1–8 + K-loop | sum, max, mean, variance, softmax | 45.7% |
| Writer | W1 | — | single output (all ops) | 99.95% |

The compiler's job is to:
1. **Select** the right reader template (const-fill, N-ary, or reduction)
2. **Set runtime args** (addresses, n_tiles, num_inputs, K)
3. **Generate** the SFPI compute kernel (the only part that varies per op)

## What's NOT covered (and shouldn't be)

- **Matmul multicast**: Matmul across a core grid needs multicast sender/receiver dataflow kernels with semaphore coordination. This is a fundamentally different pattern — not a "template" but a dedicated matmul program. It should be a separate, handwritten kernel set (or use tt-metal's existing matmul infrastructure).

- **Sharded/L1-direct**: When data is already in L1 (sharded tensors), you skip the reader entirely and point CBs directly at L1 addresses. This is a host-side configuration, not a kernel template.

- **Multi-output**: Only 2 of 4,457 kernels have 2 STOREs. Not worth a template. If needed, duplicate the writer with a second CB.

## Open questions

1. **CB reuse for >8 inputs**: The N-ary reader assumes ≤8 inputs. For fused kernels with more, the reader must multiplex CBs — read a batch of inputs, let compute consume them, then reuse CBs for the next batch. This needs compute/reader coordination beyond simple CB flow.

2. **Reduction + broadcast**: When a reduction kernel also has non-reduced inputs (e.g., `(a * b).sum(axis=0)` where b is broadcast), the reader needs mixed stride patterns — K tiles for the reduced input, 1 tile (repeated K times) for the broadcast input. The N-ary reduction reader sketch above is simplified.

3. **Tile index computation**: The readers above use linear tile indices (`0, 1, 2, ...`). For strided access patterns (transpose, slice, gather), the tile index sequence must be computed. This could be a runtime arg array or a small index-generation function in the reader. The tinygrad ShapeTracker VIEW nodes encode this — they need to be lowered to tile index sequences.
